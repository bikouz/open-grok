# Open Grok v0.1.220-open-grok.27

Open Grok is Grok Build with ChatGPT Codex, Kimi, and Fireworks AI provider
support, packaged as the independent `open-grok` command with isolated state
under `OPENGROK_HOME`.

Hotfix release: the native workflow runtime introduced in `.26` was
unusable in practice due to two stacked gating bugs. Both are fixed and
verified end-to-end with a live workflow run.

## Workflow tool restored for Codex-model sessions

The `.26` sync accidentally dropped the `workflow` tool from the
Codex-model toolset (the fork-only preset lost its single registration
during conflict deduplication). Sessions on `gpt-*` Codex models — the
most common configuration — had no workflow tool at all. It is restored,
and the preset guard test now pins `workflow` and `agent_swarm` so a
future sync cannot silently drop them.

## Remote feature flags are now ignored entirely (breaking-ish)

xAI's `cli-chat-proxy` remote settings stage upstream's own client
rollout, and a remote `workflows_enabled: false` — which xAI currently
serves — hard-disabled workflows even over an explicit
`GROK_WORKFLOWS=1`. None of those flags describe this fork's feature
set, so Open Grok now ignores the remote settings payload wholesale:

- The shell drops any remote-settings payload at session init.
- The pager no longer blocks startup (up to 2 s) joining the settings
  fetch; the prefetch still warms the model catalog and auth.
- Every feature gate — workflows, the goal family, telemetry/OTel,
  managed MCPs, auto-mode remote config, leader mode, campaigns —
  resolves from local env, `config.toml`, and fork defaults only.

If you previously relied on a remote flag to configure Open Grok,
set the equivalent in `~/.opengrok/config.toml`.

## Also

- Pager queue-drain and mermaid-cache tests are now hermetic (they no
  longer read the developer's real `~/.opengrok/config.toml` or race the
  process-global theme state).
- Test callers of a removed view accessor moved to `normal_pane_hints`.
